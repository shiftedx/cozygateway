# cozygateway wire contract v1

Status: frozen 2026-07-06. Additive changes only within v1.x. Receivers MUST ignore unknown
object fields and unknown server frame types. Unknown RichBlock types are invalid.

This document is the human-readable authority for the wire. The package
`cozygateway-contract` is its machine artifact: TypeBox schemas whose field names, types,
optionality, and constraints match every shape described here. Where this prose and the
schemas ever disagree, that is a bug in one of them; treat the two as one frozen surface.

Notation in this document:

- `field: type` is a required field.
- `field?: type` is an optional field (absent, not null, when omitted).
- `integer` is a JSON number with no fractional part.
- `A | B` is a closed union: the value is exactly one of the listed members.
- Objects are OPEN: a receiver keeps parsing when it sees a field it does not recognize.

---

## 1. Transport

Two channels:

- HTTPS REST for actions (pairing, listing, sending, configuration).
- One WebSocket per device for live traffic (streaming replies, presence, sync).

Authentication is a bearer device token. Every REST request except the two noted below
carries it:

```
Authorization: Bearer <deviceToken>
```

The WebSocket authenticates in its FIRST frame (see section 6), never in the URL or a query
string, so the token never lands in a proxy log or browser history.

`GET /health` and `POST /pair` are the only unauthenticated endpoints. Everything else
returns `unauthorized` without a valid device token.

`GET /health` returns a GatewayInfo object (section 5) so a client can confirm reachability
and read the contract version before pairing.

## 2. Content model: RichBlock

Agent-visible content is a CLOSED union of typed blocks. A conformant client renders only
these shapes. It runs no markdown parser over agent content and injects no raw HTML of agent
content, which makes the renderer the security floor.

Objects stay OPEN: a client ignores fields it does not know so that v1.x can add optional
fields. But the union of block TYPES is closed: a client that meets a block whose `type` it
does not recognize MUST treat the message as invalid rather than skip the block. A client
that cannot render a block must at least know it is looking at one.

The seven block types are:

### paragraph

```json
{ "type": "paragraph", "text": "string" }
```

- `text: string` (required).

### code

```json
{ "type": "code", "code": "string", "language": "string" }
```

- `code: string` (required).
- `language?: string` (optional).

### heading

```json
{ "type": "heading", "level": 1, "text": "string" }
```

- `level: 1 | 2 | 3` (required; exactly one of those three integers).
- `text: string` (required).

### list

```json
{ "type": "list", "items": [{ "text": "string", "checked": false }], "ordered": true }
```

- `items: ListItem[]` (required). Each ListItem is `{ "text": string, "checked"?: boolean }`.
- `ordered?: boolean` (optional).

### table

```json
{ "type": "table", "header": ["string"], "rows": [["string"]] }
```

- `header: string[]` (required).
- `rows: string[][]` (required; each row is an array of strings).

### math

```json
{ "type": "math", "latex": "string" }
```

- `latex: string` (required).

### attachment

```json
{ "type": "attachment", "fileId": "string", "name": "string", "mimeType": "string", "size": 0 }
```

- `fileId: string` (required). A gateway-scoped file identifier, NEVER a URL. No block can
  carry a navigable anchor, so a renderer cannot be tricked into following an agent-supplied
  link.
- `name: string` (required).
- `mimeType: string` (required).
- `size: integer` (required; minimum 0).

The attachment upload and download endpoints are RESERVED for a v1.x additive extension. The
reference gateway does not emit `attachment` blocks yet. A client should still parse them,
since a receiver ignores content it cannot yet fetch rather than rejecting the message.

## 3. Errors

### REST errors

A failed REST call returns a non-2xx HTTP status and this body:

```json
{ "error": { "code": "string", "message": "string" } }
```

- `error.code: string` (one of the codes below).
- `error.message: string` (human-readable, not for programmatic matching).

The frozen code list:

| code | meaning |
| --- | --- |
| `unauthorized` | missing or invalid device token |
| `not_found` | no such resource (device, thread, agent, message) |
| `invalid_request` | request body or query failed validation |
| `setup_code_invalid` | pairing setup code is wrong, used, or expired |
| `thread_archived` | the thread is archived and rejects new messages |
| `backend_unavailable` | the agent backend could not be reached |
| `turn_failed` | the agent turn started but did not complete |
| `interrupt_unsupported` | an interrupt was requested but the agent backend cannot honor it |
| `internal` | unexpected gateway fault |

Adding a code is a contract minor bump. A client treats any code it does not recognize as a
generic failure.

### WebSocket errors

A non-fatal problem on the socket arrives as an `error` frame (section 6):

```json
{ "type": "error", "code": "string", "message": "string", "threadId": "string" }
```

- `code: string`, `message: string` (required).
- `threadId?: string` (optional; present when the error is scoped to one thread).

A FATAL authentication failure does not use an error frame. The gateway closes the socket
with WebSocket close code `1008`.

## 4. Pairing

Pairing binds a new device to the gateway and mints its device token.

### The QR payload

The gateway (or its operator UI) shows a QR code carrying:

```json
{ "gatewayUrl": "https://...", "setupCode": "string" }
```

- `gatewayUrl`: the base URL the device will call.
- `setupCode`: a single-use code. It expires 10 minutes after issue and is consumed by the
  first successful `POST /pair`.

### POST /pair (unauthenticated)

Request (PairRequest):

```json
{ "setupCode": "string", "deviceName": "string", "devicePubkey": "string" }
```

- `setupCode: string` (required; minLength 1).
- `deviceName: string` (required; minLength 1, maxLength 120).
- `devicePubkey?: string` (optional).

Response (PairResponse):

```json
{ "deviceToken": "string", "device": { Device }, "gateway": { GatewayInfo } }
```

- `deviceToken: string` (required). The bearer token for all later authenticated calls.
- `device: Device` (the newly registered device; see section 5).
- `gateway: GatewayInfo` (see section 5).

A wrong, used, or expired code returns `setup_code_invalid`.

### GET /devices

Returns the paired devices as an array of Device objects (section 5).

### DELETE /devices/:id

Revokes the device with the given id immediately. Its device token stops working at once, and
any live WebSocket connection held by that device is closed. Revocation of the current device
is allowed (it logs itself out).

## 5. Resources

Shared object shapes. Timestamps are integers (epoch milliseconds).

### GatewayInfo

```json
{
  "name": "string",
  "version": "string",
  "contract": "v1",
  "capabilities": { "com.cozylabs.example": 1 }
}
```

- `name: string`, `version: string` (required).
- `contract: "v1"` (required; the literal `"v1"` for this contract).
- `capabilities?: Record<string, integer>` (optional). A map of capability id to its integer
  version. `GatewayInfo` appears in three positions on the wire: the `GET /health` response, the
  `gateway` field of PairResponse (section 4), and the `gateway` field of the `ready` frame
  (section 6); `capabilities` is part of the one shared shape, so it rides in all three.

  A gateway with no extensions to advertise may send an empty object or omit the field
  entirely; both are valid. This field is an ADDITIVE v1.x minor bump in the sense of section
  8: no existing field changes shape, and a v1.0 client that predates `capabilities` keeps
  working unmodified, because `GatewayInfo` was already OPEN. A client MUST tolerate:

  - an ABSENT `capabilities` block, from a gateway that predates this field;
  - a capability id it does not recognize, present alongside ones it does.

  In neither case does the client reject the `GatewayInfo` object or the message carrying it.

  **Vendor namespace rule.** Capability ids under the `com.cozylabs.*` prefix are CozyLabs
  vendor extensions: additional routes and frame types that ride the same transport, the same
  bearer-token auth, and the same frame envelope as the rest of this contract, but whose shapes
  are documented in CozyLabs's own docs, not here, and versioned independently of the `v1`
  contract version above. A gateway may advertise any number of `com.cozylabs.*` ids. A client
  that does not implement a given id ignores it exactly as it would any other capability id it
  does not recognize; advertising one is never a promise that a general client can use it. Other
  vendors extending cozygateway SHOULD reserve their own reverse-DNS prefix the same way, so ids
  never collide. A vendor extension graduates into this public contract only by a deliberate
  decision to move its shape into this document and freeze it as a first-class field; that
  decision, and only that decision, is what makes something part of the wire contract rather
  than a vendor's own concern.

### Device

```json
{ "id": "string", "name": "string", "createdAt": 0, "lastSeenAt": 0 }
```

- `id: string`, `name: string` (required).
- `createdAt: integer` (required).
- `lastSeenAt: integer | null` (required; `null` when the device has never connected).

### Agent

```json
{ "id": "string", "name": "string", "avatar": "string", "backend": "string", "presence": "online" }
```

- `id: string`, `name: string` (required).
- `avatar?: string` (optional).
- `backend: string` (required; identifies the agent backend).
- `presence: PresenceState` (required; one of `"online"`, `"absent"`, `"unknown"`).

### Thread

```json
{ "id": "string", "agentId": "string", "title": "string", "createdAt": 0, "lastMessageAt": 0 }
```

- `id: string`, `agentId: string`, `title: string` (required).
- `createdAt: integer` (required).
- `lastMessageAt: integer | null` (required; `null` for a thread with no messages).

### Message

A committed, durable message.

```json
{
  "threadId": "string",
  "seq": 1,
  "role": "agent",
  "blocks": [{ "type": "paragraph", "text": "..." }],
  "turnId": "string",
  "marker": "turn.failed",
  "createdAt": 0
}
```

- `threadId: string` (required).
- `seq: integer` (required; minimum 1). Per thread, gapless, starts at 1, allocated by the
  gateway in commit order.
- `role: MessageRole` (required; one of `"user"`, `"agent"`, `"system"`).
- `blocks: RichBlock[]` (required; the content, section 2).
- `turnId?: string` (optional; correlates a committed message with the turn that produced it).
- `marker?: "turn.failed" | "turn.interrupted"` (optional). `turn.failed` flags a synthetic
  system message standing in for a turn that did not finish; `turn.interrupted` flags a turn a
  user deliberately stopped (see POST /threads/:id/interrupt). Both are additive v1.x values;
  a client that predates `turn.interrupted` treats it as an unknown-but-valid system marker.
- `delivery?: "turn" | "steer"` (optional; only ever present on `role: "user"` messages). Absent
  means `"turn"`: the message started or queued its own agent turn. `"steer"` means the message
  was delivered mid-turn into an already in-flight turn instead of starting a new one. Additive
  v1.x: a client that predates the field ignores it and renders the message normally.
- `createdAt: integer` (required).

Clients dedupe by a per-thread high-water mark on `seq`.

### ToolCall

Reported inside live draft frames (section 6).

```json
{ "id": "string", "name": "string", "status": "running", "detail": "string" }
```

- `id: string`, `name: string` (required).
- `status: "running" | "ok" | "error"` (required).
- `detail?: string` (optional).

### GET /agents

Returns the agents the device can talk to as an array of Agent objects.

### GET /threads

Returns the device's non-archived threads as an array of Thread objects. An archived thread
(see DELETE below) does not appear here.

### POST /threads

Request (CreateThreadRequest):

```json
{ "agentId": "string", "title": "string" }
```

- `agentId: string` (required; minLength 1).
- `title?: string` (optional; maxLength 200).

Returns the created Thread object.

### PATCH /threads/:id

Renames a thread. Request (RenameThreadRequest):

```json
{ "title": "string" }
```

- `title: string` (required; minLength 1, maxLength 200).

Returns the updated Thread object.

### DELETE /threads/:id

Archives the thread. An archived thread disappears from `GET /threads`, but its messages stay
readable through `GET /threads/:id/messages`, and new messages are rejected with
`thread_archived`.

### GET /threads/:id/messages?before=<seq>&limit=<n>

Reads a page of committed messages. Query parameters:

- `before` (optional): a `seq`. The page returns messages strictly older than this seq. Omit
  `before` to get the newest page.
- `limit` (optional): the maximum number of messages to return.

Response (ListMessagesResponse):

```json
{ "messages": [{ Message }] }
```

- `messages: Message[]` (required). Always in ASCENDING `seq` order. With `before` and
  `limit` both set, the page is the `<limit>` NEWEST messages whose seq is below `<before>`,
  returned oldest-first. Paging backward through history means taking the lowest seq in a page
  as the next `before`.

### POST /threads/:id/messages

Sends a user message and starts an agent turn. Request (SendMessageRequest):

```json
{ "blocks": [{ "type": "paragraph", "text": "..." }] }
```

- `blocks: RichBlock[]` (required; minItems 1).

Response (SendMessageResponse):

```json
{ "message": { Message } }
```

- `message: Message` (the committed user message). The agent's reply arrives later over the
  WebSocket as draft frames and then a committed frame.

Sending to an archived thread returns `thread_archived`.

### POST /threads/:id/interrupt

Requests a hard interrupt of the thread's in-flight agent turn. Takes no request body.

Responses:

- `202 Accepted` with body `{ "status": "interrupting" }` when a turn was in flight and the
  interrupt was dispatched to the backend. For a steer-capable backend the turn ends as a
  committed system message with `marker: "turn.interrupted"` followed by a `done` frame. For a
  backend that cannot interrupt, the gateway emits an `error` frame with code
  `interrupt_unsupported` over the WebSocket and the turn runs to normal completion; the REST
  response is still `202` because a turn was in flight.
- `204 No Content` with no body when the thread has no turn in flight (a no-op).

Auth is required exactly as for every other route. A nonexistent thread returns `not_found`.

### POST /push/register

Registers a device for push notifications through a relay. Request (PushRegisterRequest):

```json
{ "pushId": "string", "relayUrl": "string", "pushKey": "string" }
```

- `pushId: string` (required; minLength 1).
- `relayUrl: string` (required; minLength 1).
- `pushKey: string` (required; minLength 1). The symmetric key the gateway uses to encrypt
  notification payloads. The relay never sees it. It travels only from device to gateway over
  the paired TLS channel.

## 6. Live stream

One WebSocket per device carries every thread. The client speaks two frame types; the gateway
speaks seven.

### Client frames (ClientFrame)

A ClientFrame is exactly one of:

**auth** (AuthFrame), which MUST be the first frame the client sends:

```json
{ "type": "auth", "token": "string" }
```

- `token: string` (required; minLength 1). The device token.

**sync** (SyncFrame):

```json
{ "type": "sync", "threads": { "<threadId>": 0 } }
```

- `threads: Record<string, integer>` (required). Maps each threadId the client cares about to
  its high-water `seq`. A value of `0` means "send everything for this thread".

### Server frames (ServerFrame)

A ServerFrame is exactly one of the seven below. The client MUST ignore any server frame whose
`type` it does not recognize; that is how the server adds frame types within v1.x.

**ready** (ReadyFrame), sent once after a successful auth frame:

```json
{ "type": "ready", "deviceId": "string", "gateway": { GatewayInfo } }
```

- `deviceId: string` (required).
- `gateway: GatewayInfo` (required).

**synced** (SyncedFrame), sent once after the gateway finishes replaying in response to a sync
frame:

```json
{ "type": "synced" }
```

**committed** (CommittedFrame), a durable message landing:

```json
{ "type": "committed", "threadId": "string", "seq": 1, "message": { Message } }
```

- `threadId: string` (required).
- `seq: integer` (required; minimum 1).
- `message: Message` (required).

**draft** (DraftFrame), an ephemeral in-progress reply:

```json
{ "type": "draft", "threadId": "string", "turnId": "string", "blocks": [ ... ], "toolCalls": [ ... ] }
```

- `threadId: string`, `turnId: string` (required).
- `blocks: RichBlock[]` (required; the current best rendering of the in-flight reply).
- `toolCalls: ToolCall[]` (required; may be empty).

A draft frame is a FULL REPLACE. Each draft for a given `turnId` supersedes the previous one
in its entirety. A draft carries no `seq`, because it is not durable.

**done** (DoneFrame), the turn's draft stream has ended:

```json
{ "type": "done", "threadId": "string", "turnId": "string" }
```

- `threadId: string`, `turnId: string` (required).

**presence** (PresenceFrame):

```json
{ "type": "presence", "agentId": "string", "state": "online" }
```

- `agentId: string` (required).
- `state: PresenceState` (required; `"online" | "absent" | "unknown"`).

**error** (ErrorFrame), described in section 3.

### Lifecycle, worked example

```
client -> { "type": "auth", "token": "<deviceToken>" }
server -> { "type": "ready", "deviceId": "dev_1", "gateway": { "name": "...", "version": "1.0.0", "contract": "v1", "capabilities": { "com.cozylabs.example": 1 } } }

client -> { "type": "sync", "threads": { "th_1": 7 } }
server -> { "type": "committed", "threadId": "th_1", "seq": 8, "message": { ... } }
server -> { "type": "committed", "threadId": "th_1", "seq": 9, "message": { ... } }
server -> { "type": "synced" }

... later, the user posts a message over REST and the agent replies live ...

server -> { "type": "draft", "threadId": "th_1", "turnId": "t_5", "blocks": [ ... ], "toolCalls": [ ... ] }
server -> { "type": "draft", "threadId": "th_1", "turnId": "t_5", "blocks": [ ... ], "toolCalls": [ ... ] }
server -> { "type": "committed", "threadId": "th_1", "seq": 10, "message": { ... "turnId": "t_5" } }
server -> { "type": "done", "threadId": "th_1", "turnId": "t_5" }
server -> { "type": "presence", "agentId": "ag_1", "state": "absent" }
```

Order within a thread: for a given turn, drafts stream, the reply commits, then `done` marks
the turn closed. `seq` on the committed message is the durable record.

### seq semantics

`seq` is per thread, gapless, starts at 1, and is allocated by the gateway in commit order. A
client keeps a per-thread high-water mark and drops any committed frame whose `seq` is at or
below it, so a redelivered commit is idempotent.

### Drafts are ephemeral

A draft frame is never durable and never carries a `seq`. Each new draft for a `turnId` fully
replaces the last. A client renders the latest draft for a turn and discards it once the turn
commits or the socket drops.

### Reconnect

On reconnect the client re-auths, then sends a sync frame with its per-thread high-water
marks. The gateway replays COMMITTED messages above each mark, in seq order, then sends
`synced`. Drafts are NOT replayed; an in-progress reply that was streaming when the socket
dropped is lost by design, and only its committed result (if the turn finished) comes back
through sync. A turn that was interrupted and never completed appears as a committed system
message (`role: "system"`) carrying `marker: "turn.failed"`, so history stays gapless and the
client can show that the turn did not finish.

## 7. Conformance

A gateway implementation is conformant when the `cozygateway-conformance` suite passes against
it while it exposes the reference echo backend.

The reference echo backend is an agent whose reply to a message whose first block is
`{ "type": "paragraph", "text": T }` is:

1. exactly two draft frames, then
2. a commit of `[{ "type": "paragraph", "text": "Echo: " + T }]`.

A value of `T` that contains the substring `[[fail]]` fails the turn: the gateway completes it
as a `turn.failed` system message rather than an echo commit.

## 8. Evolution policy

Within v1.x:

- Changes are ADDITIVE only. New optional object fields, new REST endpoints, new server frame
  types, and new error codes are all minor bumps.
- A receiver MUST ignore object fields it does not recognize.
- A client MUST ignore server frame types it does not recognize.
- A client MUST treat an unknown RichBlock `type` as invalid (the union of block types is
  closed).
- The gateway answers an unknown or malformed CLIENT frame with an `error` frame.
- A client treats an unknown error `code` as a generic failure.
- The `capabilities` block in `GatewayInfo` (section 5) is the standing mechanism for this kind
  of additive extension and for vendor namespacing; new capability ids need no version bump at
  all, additive or otherwise, since the whole point of the block is that ids come and go inside
  the same frozen `contract: "v1"`.

A breaking change (removing or renaming a field, tightening a type, changing `seq` semantics)
is a new contract version and a new value of the `contract` field in GatewayInfo.
